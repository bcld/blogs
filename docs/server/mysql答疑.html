# Mysql答疑

# mysql索引实现

  1. mysql索引实现：
    1. hash：哈希表，通过key的哈希值找到value，很快，但不支持范围查询和排序查询；时间复杂度O(1),排序会退化为O(N),树形结构永远是O(log(n))
      • 不能利用联合索引中的部分索引查询（联合索引通过合起来hash生成）
      • 不能避免表扫描，哈希碰撞时还是要比对
      • 大量hash碰撞，性能会下降
    2. b-tree：自平衡的多路搜索树，b代表balance，比起二叉树，树高度低，IO次数小
    3. bitmap，大多数不支持，mysql不支持

# b-tree和b+tree

B+树和B树的区别 - 简书 (jianshu.com) (opens new window)

  1. b树
    1. 所有结点和叶节点都存储索引和data数据
    2. 搜索可能在一个节点结束，最低复杂度为O(1)
    3. 任何一个关键字出现且只出现在一个结点
  2. b+树
    1. 所有关键字在叶子结点内出现，结点只存储关键字，叶节点存储索引和data数据，复杂度为固定log(n)
    2. 为所有叶子节点添加一个链指针，大大增加区间访问性，利用了空间局部性原理
    3. 内结点没有data，所以单次IO信息量比b-tree大

# mysql的引擎

  1. myISAM：
    1. b+tree叶节点存放数据地址，先根据索引找到对应的地址，再根据地址读取数据，称为非聚集索引
    2. 保存了count的值，所以count频繁的场景很快，适合查询频繁
  2. innoDB（必须有主键，密集索引）:
    1. 主键索引是聚集索引，主键和行记录放在同一个叶节点，找到主键即找到行记录；
    2. 非主键索引都是非聚集索引，innoDB的非聚集索引的索引文件和数据文件分开存储，叶节点保存的是主键，所以还需要根据主键去索引文件找详细行记录。主键字段不宜过长

两者的区别：InnoDB是事务型引擎，支持事务，基于聚簇索引建立（主键索引），是行锁,也支持表级锁,当不走索引的时候整张表用的是表级锁，走索引的时候用到的是行级锁； 而myISAM不支持事务（后续已经支持）和行级锁，是对整张表加锁，所有读锁释放了，写锁才可以进行，写锁是排他锁，读锁是共享锁，加了共享锁，无法再加排他锁,海量写入数据时会阻塞查询。

# 回表查询

在innoDB引擎中，非主键索引查找数据需要先找到主键，再根据主键查找具体行数据，就叫回表查询

# 问题

  1. 脏读:能读到未提交的结果 解决方法：读已提交 （Oracle默认隔离级别）
  2. 不可重复读：在自己事务提交前，多次读取结果不一致（别的事务操作提交了） 解决方法：可重复读（innordb默认隔离级别）
  3. 幻读： 解决方法：串行化

# 当前读/快照读

当前读：最新版数据 快照读: 不加锁的读，串行化下除外，（RR）快照读的时机决定读数据的版本

# explain 执行计划关键字段

列名 说明
id 执行编号，id值越大，执行优先级越高，相同id值，从上往下执行
select_type 查询的类型，主要区分是普通查询、联合查询、子查询等
type 数据访问/读取操作类型，system > const > eq_ref > ref > fulltext > ref_or_null > index_merge > unique_subquery > index_subquery > range > index > ALL ，一般来说，得保证查询至少达到 range 级别，最好能达到 ref
possible_keys 可能是用的索引
key 实际使用的索引
key_len 使用的索引的长度，长度越短越好 。
ref ref 列展示的就是与索引列作等值匹配的对象是啥。如果不是等值查询，则显示为 NULL
rows 执行计划的 rows 列就代表预计扫描的索引记录行数。
filtered 1. 对于单表，代表满足where条件的行数占查询行数百分比，意义不大。
2. 对于关联表，代表驱动表的一个扇出效率 rows*filter/100 = 扇出次数，被驱动表需要执行的次数
extra Extra 是 EXPLAIN 输出中另外一个很重要的列，该列显示 MySQL 在查询过程中的一些详细信息

  1. 访问类型type

    • const:根据主键索引 or 唯一 二级索引 与常数进行等值匹配
    • ref:使用普通二级索引和常量进行等值匹配
    • fulltext：全文索引
    • range:使用索引获取某些范围区间的记录
    • index:需要扫描全部的索引记录
    • all:全表扫描

  2. extra

    • using filesort：外部排序，无法利用索引完成排序，需要额外在内存或磁盘上排序
    • using temporary: 使用了临时表保存中间结果，常见group by / order by，一般看到说明需要进行查询优化

# SQL优化

原则

  • 最大化利用索引
  • 尽可能避免全表扫描
  • 减少无效数据查询
  1. 避免不走索引
    • 避免全模糊查询，尽量在字段后面使用模糊查询（数据量过大建议使用其他解决方案，es等）
    • 避免使用or查询条件
    • 避免在where条件等号的左侧使用表达式、函数操作
    • 不符合最左匹配原则，复合索引没有生效
    • 隐式类型转换不走索引，varchar=123
    • order by没有和where条件一致，排序不走索引
  2. 查询优化
    • 避免使用select *，按需获取列数
    • 多表关联时，小表驱动大表
