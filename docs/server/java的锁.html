# 二、乐观锁 vs 悲观锁

乐观锁：乐观认为当前资源没有被加锁，做更改哪一刻才去看有没锁 （通过版本号更新）

悲观锁：悲观认为有别的线程在竞争锁，先加锁，确保不会被别的线程修改 （显式加锁）

乐观锁主要实现：CAS (Compare And Swap) 比较与交换 ，无锁算法，底层原子操作，会循环重试，高效

CAS问题：

  1. ABA问题 通过版本号解决，1A-2B-3C
  2. 循环开销大，如果长时间不成功，会一直自旋
  3. 只能保证一个共享变量的原子操作

# 三、自旋锁 vs 适应性自旋锁

  1. 避免频繁切换处理耗费时间大于循环等待执行的时间，因此让当前线程自旋从而避免切换线程的开销，就要自旋锁，实现原理也是CAS；
  2. 在JDK 6中引入适应性自旋锁，自适应意味着自旋次数不再固定，如果某个锁经常成功，那么就会允许自旋等待较长时间，如果经常失败，那么可能直接阻塞线程，避免浪费处理器资源。

# 四、为什么synchronized能实现线程同步

背景：monitor存在每个对象的对象头中，每个对象都有一把内部锁：monitor锁，ObjectMonitor 每个线程有一个可用的monitor record列表

java对象头： synchronized是悲观锁，操作同步资源之前先加锁，锁就存在java对象头

mark word:默认存放与对象定义无关的数据：hashCode，分代年龄，锁标志位，会随着标记位变化而变化

class Point:指向类元数据的指针

**Monitor：**同步工具或同步机制，对象私有的数据结构

synchronized是通过Monitor实现线程同步，基于进入和退出monitor对象实现同步。线程执行到monitorenter指令时将会尝试获取对象对应的monitor所有权（即尝试获取对象的锁）

jdk如何对锁进行优化

  1. 锁消除：jit编译时，对运行上下文进行扫描，去除不可能存在竞争的锁。（局部方法内的synchronized）
  2. 锁粗化：通过扩大加锁的访问，避免反复加锁和解锁

# 五、无锁 VS 偏向锁 VS 轻量级锁 VS 重量级锁

锁的内存定义：当线程释放时，java内存模型会把线程对应的本地内存中共享变量刷入主内存中，而线程获取锁时，java内存模型会把该线程对应的本地内存置为无效，从而使得被监视器保护的临界区代码必须从主内存中读取共享变量。

这4种锁是指锁的状态，专门针对synchronized

  1. 无锁：
    1. 不加锁，所有线程都可以访问，但只有一个能修改成功，CAS原理及应用也是无锁的实现
  2. 偏向锁：
    1. 一段同步代码块被一个线程访问，会在对象头和栈帧中的锁记录里存储锁偏向的线程ID。锁标志位为01
    2. 用于无多线程。遇到其他线程竞争时会释放锁。
  3. 轻量级锁：
    1. 但锁是偏向锁时，被另外线程所访问，偏向锁就会升级为轻量级锁，其他线程会自旋获取锁，不会阻塞（自己），从而提高性能，锁标志位为00
  4. 重量级锁：
    1. 自旋超过一定次数，或者线程超过2条，就会升级为重量级锁，锁标志位为10。
    2. 重量级锁是将除了拥有锁的线程以外的线程都阻塞。

偏向锁升级为轻量级锁的过程：前提是处于偏向锁状态Thread-1持有，Thread-2竞争

  1. Thread-1不存在，CAS替换为Thread-2
  2. Thread-1存在，挂起Thread-1，撤销偏向锁，继续执行Thread-1，执行完后，锁升级为轻量级锁，Thread-2自旋获取锁

# 六、公平锁VS非公平锁

  1. 公平锁
    1. 多个线程按照申请顺序获得锁，线程会直接加入等待队列
    2. 等待队列中，除了第一个线程以外的都会阻塞，都需要CPU唤醒，CPU开销大
  2. 非公平锁
    1. 线程会尝试插队，CPU不需要增加唤醒工作量，吞吐量提高
    2. 如果插队失败，需要进入队列排队并阻塞，cpu需要增加唤醒工作，等同于公平锁

代码层面，两者差不多，只是公平锁在获取同步状态时多了限制条件：hasQueuedPredecessors()

# 七、可重入锁 VS 非可重入锁

可重入锁，又叫递归锁，同一个线程在外层方法获取锁后，进入内层synchronized修饰的方法自动获取锁（前提是锁对象是同一个对象或者class）。所以在一定程度上避免了死锁。

public class Widget {
    public synchronized void doSomething() {
        System.out.println("方法1执行...");
        doOthers();
    }

    public synchronized void doOthers() {
        System.out.println("方法2执行...");
    }
}

//线程调用方法doSomething后直接调用doOthers，因为内置锁synchronized是可重入的，如果不是的话，调用doOthers时锁还未释放，也获取不到锁，变成死锁。

# 八、独享锁 VS 共享锁

独享锁和共享锁同样是一种概念。

  1. **独享锁：**排他锁，该锁一次只能被一个线程持有，获取排他锁既能读也能写（synchronized和Lock都是）
  2. **共享锁：**指该锁可以被多个线程持有，如果线程T对数据A加上共享锁后，其他线程只能对A再添加共享锁，不能加排他锁。共享锁只能读数据，不能修改。
  3. 如果存在读锁，那么就会获取写锁失败，因为要保证读写一致。如果存在写锁，那么读锁要等写锁释放才能获取。

公平锁和非公平锁，只要当前线程不是拥有锁的线程都不能添加锁，所以都是独享锁
