# 浅谈Redis

remote dictionary server 是基于内存的高性能key-value的储存系统，跨平台非关系型数据库，,快，每秒可达10万次可持久化的键值对存储数据库。redis概要 (opens new window)

# 优点

  1. 读写性能优异，基于内存，单线程无锁竞争，读11/s,写8w/s
  2. 数据类型丰富 String set zset hash list，单个value最大限制是1GB，在基础数据结构之上还提供高级数据结构
  3. 可持久化（RDB和AOF）
  4. 高可用内置redis sentinel 实现主从故障自动，而且一些合并操作也是原子性
  5. 支持主从复制，可以进行读写分离

# 缺点

  1. 受物理内存限制
  2. 较难在线扩容和修改配置
  3. 引入新的组件需要维护

Redis和Memcached的区别

  1. redis支持复杂的数据结构
  2. redis原生支持集群,3.x开始
  3. redis只使用单核，memcached可以使用多核，100k以上的数据，后者性能要高
  4. 管理机制不通过，redis是现场申请内存，Memcached是预分配的内存池
  5. 网络IO模型不同，redis是单线程的IO多路复用模型，Memcached是多线程非阻塞IO复用
  6. Memcached不支持持久化，重启数据会被清空

为什么单线程性能这么高

  1. C语言实现，C语言执行快
  2. 纯内存操作，数据读到内存中，异步进行持久化
  3. 基于同步非阻塞的IO多路复用机制
  4. 单线程，避免频繁的线程切换

为什么要用redis？ 日常应用，读远多于写，数据库查询数据需要从磁盘读取，相对较慢，redis直接读内存 （高性能），速度快，同时减少数据库压力 (高并发)

# 使用redis会出现的问题

问题 原因 solve
雪崩 同一时间大量缓存失效，导致短期内全部请求数据库，造成压力激增 1. 缓存失效时间添加随机值
2. 调高hz值，提高redis主动淘汰的频率
缓存穿透 查询一个不存在缓存中的数据，数量过多同样会拖垮数据库 1. 使用布隆过滤器提前拦截
2. 把不存在的数据也放到缓存中，设置较短的失效时间
一致性问题 更新redis和数据库无法满足原子性 消息队列+重试

# 高级数据结构

  1. HyperLogLog 用于基数统计的算法，优点是节省内存空间，因为只会根据输入元素来计算基数，不存储元素本身。
  2. GEO 主要用来存储地理位置信息及操作
  3. bitmap (opens new window):BloomFilter 布隆过滤器，利用了bitMap，内存占用小，插入查询快，能够判断一定不存在，但无法判断一定存在，而且无法删除数据。可应用于解决缓存击穿和黑名单校验。
    • 布隆过滤器：当一个元素被加入集合时，通过K个散列函数将这个元素映射成一个位数组（bitmap）中的K个点，把它们置为1。插入查询时间复杂度为O(K)

# 持久化

持久化就是将数据保存到磁盘中的过程

持久化发生了什么

  1. 客户端向数据库发送写命令（客户端内存）
  2. 数据库接收到写请求（服务器内存）
  3. 数据库调用系统api写入磁盘（系统内核缓冲区）
  4. 操作系统将写缓冲区传输到磁盘控制器（磁盘缓冲区）
  5. 磁盘控制器写入到实际的物理媒介中（物理媒介）

两个备份命令：SAVR（阻塞知道rdb文件被创建完） 和 BGSAVE（fork一个子进程，不阻塞）

  • 自动触发rdb持久化方式

    • 通过配置里面SAVE 时间 次数（用的bgsave）
    • 主从复制，主往从复制
    • 执行shutdown且没有开启aof持久化

  • Redis快照-rdb，全量，实际操作时fork一个子进程处理，使用的是rdbSave（缺点，消耗性能，有丢失最新数据可能）

    • 包含两个核心函数rdbSave和rdbLoad

      • SAVE直接调用rdbSave，并在完成前阻塞调用者，期间，AOF 写入可以在后台线程进行， BGREWRITEAOF (opens new window) 可以在子进程进行
      • BGSAVE用子进程调用rdbSave，主进程仍可以继续处理命令请求
        • 原理是 fork 和 cow ：fork子进程，copy on write
      • 为了避免性能问题， BGSAVE (opens new window) 和 BGREWRITEAOF (opens new window) 不能同时执
      • 为了避免产生竞争条件， BGSAVE (opens new window) 执行时， SAVE (opens new window) 命令不能执行。
      • 调用 rdbLoad 函数载入 RDB 文件时，不能进行任何和数据库相关的操作，不过订阅与发布方面的命令可以正常执行，因为它们和数据库不相关联。

  • AOF（Append only file）仅追加文件，增量（只记录写删，查询不记录），数据太大会启用 rewrite 机制

    • 也是fork一个子进程处理
    • 子进程会把新的aof写到一个临时文件，不依赖原来的aof文件
    • 主进程持续将新的标动写入到内存和原来AOF文件中
    • 主进程获取子进程重写AOF完成信息，往新AOF同步增量变动
    • 使用新的aof文件替换原来的

bgsave原理

  • 先检查是否存在AOF/RDB子进程正在执行，存在会被拒绝执行
  • 调用rdbSaveBackground方法执行fork系统调用
  • fork用于创建进程，linux实现了cow，内核为子进程创建虚拟空间，子父共享物理空间，只有父进程发生更改时才会给子进程分配独立的物理空间

区别 快照是隔一段时间，所以数据不全，但恢复很快；而AOF是间隔很短，所以数据全，但是文件很多，恢复比较慢，恢复的过程相当于重新执行一遍命令（也不完全是，因为命令会优化，redis4.0支持混合持久化，AOF每次建立在一个快照基础上，所以不是全量。

# 分布式锁的实现

  1. 使用setnx来获取锁，设置expire参数加过期时间
    • redis支持原子操作 SET key value EX seconds NX（只适合单机）
  2. 集群情况下，slave获取同步锁信息前，master挂掉，slaver晋升为master，此时会丢失锁，使用redLock
    • 前提：5个redis实例（哨兵机制的话会比较复杂，脑裂问题）
    • 获取当前实例unix时间，ms为单位
    • 依次尝试从5个实例中获取锁，如果在响应超时时间内获取失败则失败，响应超时时间(5-10ms)应远小于锁失效时间（10s），如果获取成功，key真正有效时间 = 锁失效时间 - 获取锁耗时
    • 当且仅当从大多数实例（n/2+1）取得锁，并时间少于锁失效时间，锁才算获取成功
    • 如果因为某些原因获取锁失败（没有得到大多数，或超过有效时间）则应在所有实例上进行解锁（即便某些实例根本没有获取锁成功，防止某些节点已经获取锁只是没有来得及响应而被判断失败，导致在锁是失效内一致都不能获取锁）Redlock：Redis分布式锁最牛逼的实现 (opens new window)

redis分布式锁性能高，zookeeper分布式锁健壮性更优

# 分布式限流的实现(1小时能只能登陆5次)

使用zset实现滑动窗口限流，key value score

  1. 使用用户id+操作行为作为key
  2. 使用时间戳作为score,添加到set中
  3. 移除score 处于0到（当前时间戳-滑动周期）中的value
  4. 统计该key所有的数量，设置当前key的过期时间为周期+1s（防止网络延迟）
//拷贝自 http://svip.iocoder.cn/Redis/Interview/#Redis
public boolean isActionAllowed(String userId, String actionKey, int period,
    int maxCount) {
    String key = String.format("hist:%s:%s", userId, actionKey); // 使用用户编号 + 行为作为 KEY 。这样，我们就可以统计某个用户的操作行为。
    long nowTs = System.currentTimeMillis(); // 获取当前时间。
    Pipeline pipe = jedis.pipelined(); // pipeline 批量操作，提升效率。
    pipe.multi(); // 此处启动了事务，可以保证指令的原子性。
    pipe.zadd(key, nowTs, "" + nowTs); // zset 添加，key value score 要看下。
    pipe.zremrangeByScore(key, 0, nowTs - (period * 1000)); // zremrangeByScore ，移除超过周期的 value 。

    Response<Long> count = pipe.zcard(key); // zcard ，计算 zset 的数量
    pipe.expire(key, period + 1); // 设置过期。这里多 + 1 秒，为了防止网络延迟。
    pipe.exec(); // pipeline 执行
    pipe.close();

    return count.get() <= maxCount; // 是否超过最大次数。
}

# 实现消息队列

  1. 使用list结构作为队列，rpush生产消息，lpop消费，缺点是不会等待有消息才消费，会一直消费，可以引入sleep机制去调用lpop重试
  2. 也可以使用blpop key timeout，会阻塞队列直到有消息或者超时，缺点是只能供一个消费者消费
  3. 使用pub/sub主题订阅模式可以实现1:N的消费队列，缺点是发布是无状态，无法保证到达，消息即发即逝，所以消费者下线情况下会丢失消息

# 主从同步原理

  • 支持主从复制（单向流动）和从从复制（新增）
  • 主节点挂了，从节点提供服务，保证高可用
  • 读写分离，主读写，从只读，一个从只能有一个主
  • 第一次同步，主做bgsave操作，同时把后续修改先记录到内存buffer中，等数据快照同步到slave后再将期间的操作记录回放到复制节点

主从同步的弊端是不具备高可用，因为master挂掉后，redis将不能提供写入操作，所以有了哨兵机制

哨兵机制（redis Sentinel），本身也是一个独立运行的进程

  • 哨兵有一个节点或多个节点组成，特殊的节点不存储数据
  • 主节点和从节点都属于数据节点 功能：
    • 监控+自动故障转移（选取新的主节点）
    • 配置提供（提供数据节点地址）+通知（故障转移结果）
    • 转移的规则
      • 淘汰ping时间超过5s
      • 淘汰超过连接时间阈值的
      • 选取复制偏移量最大的，同等下选id最小的机器

# redis集群原理

背景：如何在海量数据中快速查找所需要的数据

  1. 无中心结构，每个节点保存数据和整个集群的状态，每个节点都和所有节点连接
  2. 使用gossip协议（流言协议，反熵思想）传播信息以及发现新的节点
  3. 数据分片，通过hash()函数去对节点数求模，以达到均匀分布的目标，缺点是不可以动态调整节点数
    • 解决方案：
      • 一致性哈希算法，对2^32求模（哈希算法返回int整数类型，32位），将哈希值空间组织成虚拟的圆环，0-2^32-1
      • 数据根据一样的哈希函数运算后求模，顺时针第一台服务器就是目标存储服务器
      • 对于节点的增减都只需要重新定位一小部分的数据，具有较好容错性和扩展性
      • 缺点是：数据倾斜问题，数据不是均匀分布。（解决方法引入虚拟节点解决数据倾斜问题。将虚拟节点平均分在圆环上，对虚拟节点进行映射）

# redis6.0新特性--多线程

  1. 之前的版本也不是完全单线程，除了主线程，也有后台线程在处理一些缓慢的操作。
  2. redis瓶颈主要在于网络的IO消耗，利用多线程分摊IO读写负荷
  3. 不会存在并发问题，应为多线程部分只是用于处理网络数据的读写和协议的解析，命令执行依旧是单线程顺序执行。

# 查询固定前缀的key

从海量key中查询某一固定前缀的key（得看数据量）

  • keys pattern:keys指令查找匹配的key
    • 一次性返回，阻塞
  • scan cursor [MATCH pattern] [COUNT count]
    • 非阻塞，游标从0到0返回作为一次完整的查询
    • 指定数据返回的范围，每次返回数据量不一定等于指定值，通过游标分多次查询
    • 获取的游标不一定一直递增，所以有重复数据，需要在外部逻辑去重处理

# 开发工具

  • Jedis：对Redis命令的封装，通过java连接redis的工具包，提供redis各种命令支持 api
  • spring-data-redis：spring对redis的整合，对redis底层开发包(jedis/Lettuce)进行了高度封装，RedisTemplate提供了redis各种操作
    • 2.0版本之前setnx和expire两个命令非原子性
    • 2.0版本之后，Redis通过set(key,value,expire)命令，将setnx和expire两个命令合并为一个原子操作
    • 对于耗时的业务需要开启一个守护线程（new Thead().setDaemon(true)），定时给锁续约
    • 单机模式（指redis）下CP模型，集群模式下AP模型，需要redLock等算法保证分布式锁
  • redLock：分布式锁的一种实现算法，用于多节点redis实现分布式锁
  • Redisson：在redis基础上，基于netty通讯的客户端，有一整套成熟的分布式解决方案，自动续约（每1/3的超时时间触发）和redLock的实现

# redis常见问题

  1. master最好不要做持久化，特别是内存快照持久化
  2. 主从最好在同一个局域网内
  3. 避免在压力较大的主库上增加从库
  4. 为了master稳定性，主从复制不要使用图状结构，而是单向链表更稳定，替换也更快
  5. 快速找出redis上以某个固定值作为前缀的key，keys指令最快但阻塞，scan指令无阻塞，但慢点有一定概率重复性，可以客户端再做去重处理。
