# 底层存储方式

磁盘:外部存储器的介质,磁盘通过顺序读写可以实现高吞吐.

页缓存page cache:磁盘的一种缓存,减少I/O次数

  • 写入的时候其实是写到页缓存中,小写合并成大写后刷盘,即后写,避免断电导致未刷盘的脏页丢失,可以使用fsync强制刷盘,性能损耗较大
  • 读会预读,会把相邻的磁盘读入页缓存中.
  • 一般建议多副本机制保证消息可靠,而不是同步刷盘

mmap:文件映射,将程序虚拟页面直接映射到页缓存上，这样就无需有内核态再往用户态的拷贝，而且也避免了重复数据的产生

DMA:全称 Direct Memory Access ，它可以独立地直接读写系统内存，不需要 CPU 介入，像显卡、网卡之类都会用DMA

# 普通读写流程

4次上下文切换+2次拷贝(3个地方-缓冲区\页缓存\socket Buffer)

# mmap读写(mmap+write)

4次上下文切换+1次拷贝(2个地方-页缓存\socket Buffer)

# sendfile(linux2.4+)

2次上下文切换+0次拷贝

# rocketMQ储存方式(采用mmap,还有一步:页缓存到socket buffer的拷贝)

  • 采用Topic混合追加方式，即一个 CommitLog 文件中会包含分给此 Broker 的所有消息，不论消息属于哪个 Topic 的哪个 Queue 。
  • 消息顺序写入,追加到commitLog中
  • 建立消息对应的consumerQueue,存储对应的物理地址,相当于索引

rocketmq做了一些优化:

  1. 文件预分配,会提前准备好下一个文件的分配，防止在消息写入的过程中分配文件，产生抖动。
  2. 文件预热,减少大文件 mmap 因为缺页中断产生的性能问题

Summary

  1. 顺序写盘，整体来看是顺序读盘，并且使用了 mmap代替read()，不是真正的零拷贝
  2. 由于页缓存不稳定性和mmap惰性加载(缺页中断才真正加载数据),用文件预先分配和文件预热优化

# Kafka储存方式

  • 一个分区一个文件,迁移复制灵活(易扩展),但数量多会增加切换成本
  • Kafka 的消息写入对于单分区来说也是顺序写，如果分区不多的话从整体上看也算顺序写
  • 发消息 Kafka 用到了零拷贝(代码调用FileChannel.transferTo,底层就是 sendfile)
