# RAFT

Raft是一种共识算法（consensus algorithm），也是实现分布式一致性的协议。

在分布式容错系统中,不同服务器就取值问题达成一致是一个根本的问题。

F：followers 跟随者 (只读服务器节点)
C：candidate 候选人 (只读服务器节点,可竞选为领导者)
L：leader 领导者 (读写服务器节点)

# 大致流程

  1. C发起(拉票)请求给其他节点F

  2. 其他节点F响应(投票)

  3. 如果某个C节点拿到大部分的票,升级为领导者L

  4. 所有的值变更都需要经过领导者(可写)

  5. 领导者传递变更给其他节点F同步，直到大部分节点响应成功，才提交值变更

  6. 值变更提交后,L通知给其他节点F

    C节点成为候选人（投票给自己），接单AB收到C的拉票并响应给C

# Leader Electiton

  • 与选举相关的超时设置有两个
    • 竞选时长(election timeout)
    • 心跳时长(heartbreak timeout)
  1. 跟随者F升级为候选人所需等待的时间为竞选时长，随机150ms-300ms
  2. F升级为候选人C后，开启竞选期,发起(拉票)请求给其他节点F，如果收到请求的节点在竞选期内并没有给别的候选人投票（响应）则会响应给当前拉票人，并且重置F自己的竞选期，
  3. 一旦某个候选人C拿到大多数票，就会在心跳期间(heartbreak timeout)发送追加条目信息(Append Entries Messages)给追随者F，追随者接收后会重置自己的竞选期
  4. 因此如果一直没有收到心跳，超过追随者的竞选期(主要收到就会重置竞选期)，追随者会成为候选人Candidate

# Split vote

  1. 当超过一个候选人同时具有成为领导者资格，就会出现分票的情况
  2. 这两个候选人就会同时重新进入下一个选举期，由于election timeout是随机的，因此必然会有先后关系，从而只会有一个成为领导者

# Log Replication(日志拷贝，保持一致)

  1. 一旦选出一个领导者，需要拷贝同步日志给其他跟随者，这一过程通过用于传递心跳的追加条目信息实现
  2. 客户端发送信息(譬如赋值)给服务器，会追加到日志里，会在下一次心跳触发时发给追随者
  3. 一旦得到大多数追随者响应确认，就会提交该操作，返回响应给客户端，且通知F
  4. 如果进行了分区，互不同，则会出现多个领导者（不同election term的，参照Split vote不可能在同一term）
  5. 此时如果客户端对两个领导者分别发送信息，那么非最新的term将不能得到大多数追随者的响应确认，因此会保持未提交状态；相反最新term的领导者能成功提交信息
  6. 修复分区后，非最新term的领导者将会发现存在最新领导者，因而会暂停，此时其以及其追随者都会回滚未提交信息，重新匹配最新领导者的日志
  7. 这样日志就能在分区中保持一致性。 AB节点在一个分区分区，任期为1落后于BCD所在的分区，领导者B接单拿不到大多数响应，因此无法提交log，而C则可以
