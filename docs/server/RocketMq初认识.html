rocketMq是一个消息中间件，常用于处理系统业务消息。实现系统之间解耦、异步消息、削峰填谷、分布式事务消息，底层采用Netty NIO框架实现数据通信。目前常见的框架还有RabbitMq（性能相对比较差，阻塞时性能急速下降）,Kafka（吞吐量高，但是主题多的话，实际吞吐量会下降，而且不排除有丢失，所以常用于日志平台）。

# 基本构成

主要由broker(消息中转角色，负责储存和转发，定时发送心跳包，和nameSrv保持长连接)，producer，consumer，nameServer（相当于路由，存储topic和broker关系）组成。

如何配置Namesrv地址到生产者和消费者？

  • 编程方式，就像 producer.setNamesrvAddr("ip:port") 。
  • Java 启动参数设置，使用 rocketmq.namesrv.addr。
  • 环境变量，使用 NAMESRV_ADDR。
  • HTTP 端点，例如说：http://namesrv.rocketmq.xxx.com 地址，通过 DNS 解析获得 Namesrv 真正的地址。

消费模式：

  • 集群消费（多个consumer示例平均消费topic对应的队列），返回失败会重试
  • 广播消费(每个consumer全量消费topic对应的队列)，不可重试

topic:表示信息第一级别，必填，一般按业务系统分。 Queue:子主题，相当于对应不同的业务功能

消息的种类：

  1. 异步消息，不等待响应，通过回调接口接收服务器响应
  2. 同步消息，会阻塞MQ服务直到返回响应消息，常用于邮件短信消息
  3. 单向消息，不等待回应且没有回调函数触发，用于日志

# 消息存储原理

基本过程

  1. 生产消息时将消息存在commitLog中

  2. 读取时，先读取ConsumeQueue(有索引那味)，再通过ConsumeQueue中的物理位置信息获取CommitLog中具体的消息数据

  • 随机IO Broker单个实例下所有的队列公用一个日志数据文件(commitLog)占用内存,而Kafka采用独立型的存储结构，每个队列一个文件，类型多的时候，其实顺序IO会变成随机IO（每个文件内是顺序，但需要在每个文件中切换，所以对于系统来说相当与随机），所以性能下降明显。

    存储介质的性能很大程度依赖于数据被访问的模式，SATA硬盘，随机I/O性能比顺序I/O性能慢3-4个数量级，SSD相差不大

  • 零拷贝技术 减少了1次CPU拷贝，提高了效率能，非真正意义的零拷贝(无cpu零拷贝)，使用mmap()代替read()，减少了一次cpu的拷贝。代价是映射操作需要虚拟存储操作开销。kafka是真正的零拷贝，效率更高。

# 重复消费

可能的原因：

  1. 调用链路过长，上游重试
  2. 发送超时，导致重试
  3. 消费者消费后宕机，没有回传offset给MQ，导致重发消息给消费者

解决方案：

  • 生产端
    • RocketMq支持消息查询功能，可以查询是否已发送（高并发下影响性能）
    • 引入redis，缓存发送的数据，查询是否已发送（redis维护）
  • 消费端
    • 构建中间表引入唯一索引，入库时校验
    • 消费端的业务逻辑保持幂等性

# 发送事务消息

保证本地事务和消息发送结果一致（非消费结果）

  1. rocketMq发送prepared信息，拿到消息的地址
  2. 执行本地事务
  3. 通过步骤1那到的地址访问消息，并修改状态
  4. rocketMq会定期扫描集群中信息，发现prepared消息，会向消息发送者确认，根据策略来决定是回滚还是继续发送确认消息

# 消费事务消息

消费失败和消费超时怎么解决？

  1. 消费超时则不断进行重试
  2. 消费失败则人工介入解决，边际效益，成本太高

# 消息丢失

  1. 丢失的场景

    • 发送mq时网路抖动，没发送成功
    • 持久化过程中宕机导致消息丢失
    • 消息还没消费就通知消费完，结果系统宕机了

  2. 解决方案

    • 通过事务来确保消息成功发送，首先发送half信息，待事务执行成功提交half消息
    • 采用同步刷盘策略，集群部署，多个follower备份
    • 消费者使用确认机制，如果没success就重新交给其他消费者消费

rocketMQ强烈建议分布式集群保证高可用
